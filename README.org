#+TITLE: clutch — Interactive Database Client for Emacs
#+AUTHOR: Lucius Chen

* Overview

An Emacs database client with an interactive data browsing UI.
Supports MySQL, PostgreSQL, and SQLite via pure Elisp backends, plus Oracle,
SQL Server, DB2, Snowflake, and Amazon Redshift through a lightweight JVM
sidecar (=clutch-jdbc-agent=, Java 17+).

- =clutch-db.el= — generic database interface (=cl-defgeneric= dispatch)
- =clutch-db-mysql.el= — MySQL backend adapter
- =clutch-db-pg.el= — PostgreSQL backend adapter
- =clutch-db-sqlite.el= — SQLite backend adapter (uses Emacs 29.1+ built-in sqlite)
- =clutch-db-jdbc.el= — JDBC backend adapter (Oracle, SQL Server, DB2, Snowflake, Redshift, and any JDBC-compatible database)
- =clutch.el= — interactive UI layer: single-page result browser, record view, FK navigation, editing, transient menus, REPL
- External native protocol dependencies:
  - =mysql= — pure Elisp MySQL wire protocol client
  - =pg= — PostgreSQL client from =pg-el=

For Org-Babel integration, install the separate =ob-clutch= package.

* Features

** Native Backends

- MySQL, PostgreSQL, and SQLite are supported without a sidecar process
- MySQL and PostgreSQL use pure Elisp protocol implementations
- SQLite uses Emacs 29.1+ built-in =sqlite-*= functions
- For native backend protocol, TLS, timeout, and connection examples, see
  [[file:docs/native-backends.md][docs/native-backends.md]]

** Interactive UI (clutch.el)

- Single-page result display with horizontal scrolling for wide tables
- SQL pagination (=LIMIT/OFFSET=) with fixed header and status bar
  - =mode-line=: row count, page, elapsed time, sort state, staged-change summary, aggregate status, row-identity guidance, active filters, and cursor position (=R-N:column-N/M=) (with =nerd-icons=, Unicode fallback)
  - =header-line=: column names with =│= borders and underlined labels
- After execution, focus stays in the SQL buffer while results refresh below
- Last successfully executed SQL is marked at the statement start gutter in the editor
- Row numbers as a table column; sort indicators via =nerd-icons= (with Unicode fallback)
- Right-aligned numeric columns
- Record buffer for single-row detail view
- Foreign key detection and navigation
- Row-identity-backed inline cell editing with staged UPDATE generation; editable results use primary keys first, then non-null unique keys, then backend row locators when available
- Staged INSERT (=i=), DELETE (=d=), and staged SQL preview/commit
- WHERE filtering (=W=), client-side fuzzy filter (=/=), SQL =ORDER BY= sorting (=s=)
- Column-aligned horizontal paging via =[= / =]=, plus column width adjustment via =-= / ===
- CSV / INSERT / UPDATE export, plus TSV / CSV / INSERT / UPDATE copy
- Agent context copy (=clutch-copy-context-for-agent=): copy the current SQL,
  connection context, referenced table metadata, and the visible columns from
  the latest matching result sample as Markdown for external tools such as
  ChatGPT, Claude, or DeepSeek
- Object picker and actions for =Tables= / =Views= / =Synonyms= / =Indexes= / =Sequences= / =Procedures= / =Functions= / =Triggers=, with annotated minibuffer candidates, type-grouped completion, optional prefix filtering, and a no-Embark transient fallback
- SQL completion: table/column names scoped to the current statement, with native MySQL/PostgreSQL hot paths staying cache-first and scheduling idle-time column preheat on cache miss instead of blocking typing; JDBC backends keep their direct metadata path; completion case can be customized via =clutch-sql-completion-case-style=
- Eldoc: three-tier lookup — (1) schema-aware: table name shows database, column count, and table comment; column name shows =table.column=, type, nullability, PK/FK references, and comment; schema metadata is cache-first and quietly preheated on the main thread once Emacs is idle, so point motion does not block; (2) live MySQL =HELP= lookup (MySQL only, result cached per session); (3) static reference covering ~150 SQL keywords and functions with call signatures and descriptions (works without a connection; MySQL/PG-specific entries annotated)
- Mode-line spinner (=…=) while a query is executing, so you always know when clutch is waiting for the server
- Dedicated query console per connection (=clutch-query-console=), with fast switching between open consoles (=clutch-switch-console=); console content persisted across Emacs sessions
- Schema refresh state is explicit: MySQL, PostgreSQL, and JDBC start a deferred schema snapshot after connect/reconnect, while SQLite keeps its lightweight synchronous path; query console buffer names surface =schema...= / =schema~= / =schema!= / =schema 42t= style state markers; here =Nt= means "the current schema-cache snapshot contains N browseable table/object names". =schema 0t= is therefore a successful empty snapshot, not an error. Schema errors also appear in the console header-line (see below). Manual =C-c C-s= refresh is explicit and synchronous, so it can recover a failed deferred refresh immediately.
- Native MySQL/PostgreSQL also preheat passive metadata via idle-time callbacks on the main thread: result buffers render immediately and then enrich cached column details, while object warmup keeps non-table categories off the first-open path without creating Emacs worker threads
- Console header-line shows live connection context: database-type icon + backend name, a dedicated connection-state icon before the compact connection key (=user@host=, with =:port= only when non-default), a separate namespace segment with =nf-md-sitemap_outline= + the current schema/database whenever the effective namespace is known, and, for connections that support Clutch manual commit, =Tx: Auto= / =Tx: Manual= / =Tx: Manual*= for the current commit mode; SQLite omits the transaction segment. Disconnected state keeps the backend context visible and warns explicitly instead of dimming it away
- Cache-backed schema actions also warn when the schema cache is stale, failed, or still refreshing, so completion / table prompts do not silently pretend cached metadata is current
- Auto-reconnect: if the connection drops, clutch silently reconnects before the next query using the stored credentials
- Value viewers: =v= opens a static one-shot viewer for the current cell; =V= opens a live viewer that follows point across result / record buffers.  Both auto-detect JSON (pretty-printed with =json-ts-mode=), XML (formatted via =xmllint= if available, rendered with =nxml-mode=), or BLOB (concise text/hex preview); XML/BLOB size is shown in the viewer header-line (without polluting content highlighting)
- Per-table/per-column result displayers can override cell text via =clutch-register-column-displayer= or =clutch-column-displayers= while keeping the raw cell value, value viewers (=v= / =V=), and cell face logic unchanged
- REPL mode (comint-based)

Row identity support for staged =UPDATE= / =DELETE=:

| Backend | No-primary-key support |
|---------+------------------------|
| MySQL native | non-null unique keys; no physical row-locator fallback |
| PostgreSQL native | non-null unique keys, then =ctid= for ordinary heap tables |
| SQLite native | non-null unique keys, then =rowid= for tables not declared =WITHOUT ROWID= |
| Oracle JDBC | non-null unique keys, then =ROWID= |
| Other JDBC backends | non-null unique keys only |

Joined, grouped, derived, or otherwise ambiguous result sets remain read-only
for staged =UPDATE= / =DELETE= unless clutch can identify a single source table
and inject or read a matching row identity.

* Requirements

- Emacs 29.1+
- MySQL 5.6+ / 8.x, PostgreSQL 12+, or SQLite 3 (via Emacs built-in)
- MariaDB 10.11 has been live-validated through the native =mysql= backend;
  TLS with =mysql_native_password= has also been live-validated; other
  MariaDB versions are expected to be broadly compatible but are not yet part
  of the regular test matrix
- Java 17+ for the JDBC backend (Oracle, SQL Server, DB2, Snowflake, Redshift)

For MySQL-family servers, core functionality (connect/query/object workflow,
editing, prepared statements, and clutch UI workflows) is expected to work on
MySQL 5.6+ and on compatible MariaDB releases.  Some newer server-side
features remain version-specific:

- MySQL 5.6 does not provide a native JSON column type
- MariaDB exposes =JSON= as an alias over text with =json_valid(...)= checks
- MariaDB 10.11 TLS has been validated with =mysql_native_password=; the older
  =mysql_old_password= plugin is not supported by the native client
- newer auth plugins and TLS combinations vary by server family/version

* Installation

Install =clutch= from MELPA or another package archive:

#+begin_src emacs-lisp
(package-install 'clutch)
#+end_src

Package-manager installs also install the core native backend protocol
packages:

- =mysql= for =:backend mysql=
- =pg= for =:backend pg=

SQLite support is built in on Emacs 29.1+, and JDBC support ships inside
=clutch= itself (the runtime still needs Java 17+ and =clutch-jdbc-agent.jar=).
For Org-Babel support, install the separate =ob-clutch= package.

For source checkouts, add =clutch= and any native protocol backend checkout you
use to =load-path= explicitly:

#+begin_src emacs-lisp
(add-to-list 'load-path "/path/to/clutch")
(add-to-list 'load-path "/path/to/mysql.el") ; only for :backend mysql
(add-to-list 'load-path "/path/to/pg-el")    ; only for :backend pg
(require 'clutch)
#+end_src

* Upgrading From Pre-Split Checkouts

This release separates protocol and Org-Babel packages from the main
=clutch= package.  If you are upgrading from an older checkout that bundled
everything in one repository, this is a breaking packaging change:

- =mysql.el= is no longer bundled in this repository.  Package-manager installs
  of =clutch= install the =mysql= package as a dependency; source checkouts must
  add =mysql.el= to =load-path= for =:backend mysql=.
- =pg.el= is no longer bundled in this repository.  Package-manager installs of
  =clutch= install the =pg= package as a dependency; source checkouts must add
  =pg-el= to =load-path= for =:backend pg=.
- =ob-clutch.el= is no longer bundled; install =ob-clutch= separately for
  Org-Babel source blocks.
- Saved connections must specify =:backend= explicitly.

* Interactive Client

** Quick Start

*** 1. Configure connections

#+begin_src emacs-lisp
(add-to-list 'load-path "/path/to/clutch")
(require 'clutch)

(setq clutch-connect-timeout-seconds 10
      clutch-read-idle-timeout-seconds 30
      clutch-query-timeout-seconds 20
      clutch-jdbc-rpc-timeout-seconds 15)

(setq clutch-connection-alist
      '(("dev-mysql"  . (:backend mysql
                          :host "127.0.0.1" :port 3306
                          :user "root" :password "test"
                          :database "mydb"
                          :connect-timeout 5
                          :read-idle-timeout 60))
        ("dev-pg"     . (:backend pg
                          :host "127.0.0.1" :port 5432
                          :user "postgres" :password "test"
                          :database "mydb"))
        ("prod-pg"    . (:backend pg
                          :host "pg.internal" :port 5432
                          :user "app" :database "appdb"
                          :ssh-host "bastion-prod"))
        ("remote-pg"  . (:backend pg
                          :host "127.0.0.1" :port 55433
                          :user "app" :database "appdb"
                          :tramp "/ssh:devbox:/workspace/"))
        ("dev-sqlite" . (:backend sqlite
                          :database "/path/to/my.db"))))
#+end_src

- Every entry must specify =:backend= (=mysql=, =pg=, =sqlite=, or a JDBC driver
  symbol such as =oracle=).
- SQLite is local and file-based: only =:database= is required; use =":memory:"=
  for a transient database.  Do not combine SQLite with =:ssh-host= or =:tramp=.
- =:password= is optional for network backends; see [[*Password Management][Password Management]] for
  auth-source integration.
- Network backends accept =:connect-timeout= and =:read-idle-timeout=.  JDBC also
  accepts =:query-timeout= and =:rpc-timeout=.
- TLS can be enabled with =:tls t=.  For explicit plaintext, prefer
  =:ssl-mode disabled= on MySQL and =:sslmode disable= on PostgreSQL.
- =:ssh-host= and =:tramp= both create local OpenSSH forwards for structured
  =:host= / =:port= endpoints; =:ssh-host= uses a =~/.ssh/config= host alias,
  while =:tramp= derives the SSH target from an ssh-like TRAMP directory.
  =:tramp-default-directory= remains accepted as a longer spelling.
- Query-console SQL is saved by connection identity rather than saved connection
  name, so renaming a saved connection keeps the same console SQL.

JDBC backends use a driver symbol as =:backend=.  See
[[file:docs/jdbc-backend.org][docs/jdbc-backend.org]] for supported databases and setup.

*** 2. Open a query console

#+begin_src text
M-x clutch-query-console     ;; Select a saved connection, or press RET on an
                              ;; unmatched connection to create a temporary connection
M-x clutch-query-sqlite-file ;; Shortcut: select a SQLite file → opens a connected SQL console
                             ;; Header-line: MySQL[root@127.0.0.1:3306/mydb]
                             ;;         or PostgreSQL[postgres@127.0.0.1:5432/mydb]
                             ;;         or SQLite[my.db]
#+end_src

Repeated calls with the same name switch to the existing buffer instead of
opening a new one.  Use =clutch-switch-console= to jump between open consoles.
=clutch-query-console= lists saved connections; pressing RET with no matching
connection starts a temporary connection flow and opens a query console for it.
SQLite is part of that flow and asks for a database file instead of host/user
credentials.  =clutch-query-sqlite-file= is the direct shortcut for local
SQLite files.  If no saved connections are configured, =clutch-query-console=
starts the temporary connection flow immediately.

*** 3. Write and execute SQL

#+begin_src sql
SELECT * FROM users LIMIT 10;
#+end_src

Press =C-c C-c= to execute.  If a region is selected, the selected SQL runs; otherwise the statement at point runs.  Results appear in a split result buffer below.

*** 4. Control transactions

For native MySQL and PostgreSQL, and for JDBC connections that run in
manual-commit mode, clutch uses the same transaction keys:

- =C-c C-a= toggles auto-commit when supported
- =C-c C-m= commits
- =C-c C-u= rolls back

SQLite does not expose these transaction controls; its console header omits the
=Tx= segment.

Native MySQL maps =C-c C-a= to the server session's autocommit flag directly.
Native PostgreSQL uses a clutch-managed manual mode: enabling manual mode does
not send =BEGIN= immediately, the first foreground statement opens the
transaction lazily, and transactional DDL also counts as uncommitted work, so
toggle/disconnect stays blocked until you commit or roll back.

** Password Management

=:password= is optional in connection entries. When omitted (or nil), the
password is looked up via Emacs's built-in =auth-source= library, which supports:

- =~/.authinfo= / =~/.authinfo.gpg= (netrc format)
- =pass= (the Unix password manager), when =auth-source-pass= is enabled

*** ~/.authinfo example

#+begin_src text
machine 127.0.0.1 login root password secret port 3306
#+end_src

Connection entry without hardcoded password:

#+begin_src emacs-lisp
("dev-mysql" . (:backend mysql
                :host "127.0.0.1" :port 3306
                :user "root" :database "mydb"))
#+end_src

*** pass example — connection name as entry name (recommended)

When =auth-source-pass= is loaded, clutch automatically looks up a pass
entry whose path ends with the connection name.  The entry can live
anywhere in the pass store — subdirectories are fine:

#+begin_src emacs-lisp
(require 'auth-source-pass)

(setq clutch-connection-alist
      '(("dev-mysql" . (:backend mysql
                         :host "db.example.com" :port 3306
                         :user "alice" :database "mydb"))))
#+end_src

Store the password under any path that ends with the connection name:

#+begin_src sh
pass insert mysql/dev-mysql   ;; or just: pass insert dev-mysql
#+end_src

The pass store reveals only =mysql/dev-mysql= — no host or username.
Use =:pass-entry "other-name"= to override the suffix when the entry
name must differ from the connection name.

For JDBC connections (including Org-Babel blocks), an explicit =:pass-entry=
that resolves to no password now fails fast in Emacs instead of sending a null
password to the driver.  If you use =pass=, make sure =auth-source-pass= is
enabled first.

*** Fallback: standard auth-source lookup

If no pass entry matches the connection name, clutch falls back to
standard =auth-source-search= by =:host=, =:user=, and =:port=.  This
covers =~/.authinfo=, =~/.authinfo.gpg=, and pass entries named after
the host.  See the [[https://www.gnu.org/software/emacs/manual/html_node/auth/index.html][auth-source manual]] for details.

** SSH Tunnels via ~/.ssh/config

For bastion-style access, add =:ssh-host= to a saved network connection:

#+begin_src emacs-lisp
("prod-pg" . (:backend pg
               :host "pg.internal" :port 5432
               :user "app" :database "appdb"
               :ssh-host "bastion-prod"))
#+end_src

clutch then starts a local SSH tunnel with your normal OpenSSH setup and
connects the database client through the forwarded local port automatically.
The SSH authentication itself is expected to come from your existing
=~/.ssh/config=, =ssh-agent=, host aliases, =ProxyJump=, and known_hosts.

If the first connection still needs a key passphrase or host-key confirmation,
run =M-x clutch-prepare-ssh-host= once for that host alias, or press =S= from
the =C-c ?= transient menu.  That opens a normal interactive =ssh= session and
leaves =clutch-connect= on the usual non-interactive path afterward.

The batch tunnel does not prompt for SSH credentials.  If interactive =ssh HOST=
succeeds but clutch still reports SSH authentication failure, make sure your
OpenSSH setup can authenticate non-interactively too, for example via
=ssh-agent= and =AddKeysToAgent=.

This keeps the Emacs flow simple:

- configure SSH once in OpenSSH
- keep database passwords in =auth-source=
- connect with the usual =M-x clutch-connect= or =M-x clutch-query-console=

Current limitation: SSH tunneling is only wired up for structured
=:host= / =:port= entries, so JDBC profiles that rely on a raw =:url= still
need a manual tunnel for now.

** TRAMP-aware Connection Origin

Opening a TRAMP buffer does not globally change clutch.  TRAMP is considered
only when =clutch-connect= or =clutch-query-console= creates a connection, and
the chosen origin is stored so reconnect, completion, refresh, and query
execution keep using the same route.

For a saved connection that should always use a remote-machine endpoint,
configure =:tramp= explicitly.  The =:host= / =:port= pair is the database
endpoint as seen from that SSH host:

#+begin_src emacs-lisp
("remote-pg" . (:backend pg
                 :host "127.0.0.1" :port 55433
                 :user "app" :database "appdb"
                 :tramp "/ssh:devbox:/workspace/"))
#+end_src

Without =:ssh-host= or =:tramp=, commands invoked from a TRAMP buffer can infer
that buffer's remote context.  =clutch-tramp-context-policy= controls this:

- =nil=: never infer TRAMP context
- =ask= (default): ask before using the current TRAMP context
- =auto=: use the current TRAMP context without asking

Supported and unsupported cases:

- Supported TRAMP methods are ssh-like paths: =/ssh:=, =/scp:=, =/rsync:=, and
  =/rpc:= when using tramp-rpc.  SSH-like hop chains are mapped to OpenSSH
  =ProxyJump=.  Other TRAMP methods are not inferred from the source buffer.
- Clutch uses the local OpenSSH client; it does not require =nc=, =netcat=, or
  =socat= on the remote host.
- TRAMP forwarding is still an OpenSSH batch forward.  A TRAMP buffer opened by
  typing a password does not automatically lend that password to clutch; use
  non-interactive OpenSSH auth or a reusable ControlMaster.  When tramp-rpc has
  an active ControlMaster, clutch can reuse its ControlPath for =/rpc:=.
- Container TRAMP paths such as =/docker:user@container:/workspace/= or
  =/ssh:devbox|podman:app:/workspace/= are not enabled yet.  They need a
  separate container transport because OpenSSH =-L= cannot target a container
  namespace directly.
- Forwarding only applies to structured =:host= / =:port= entries.  SQLite is a
  local file open, and raw JDBC =:url= profiles are not rewritten.

** Working with SQL files

You can also open any =.sql= file, enable =clutch-mode=, and connect
manually — the query console is not required.

#+begin_src text
1. Open a .sql file
2. M-x clutch-mode        — enable clutch (inherits sql-mode syntax/fontification)
3. C-c C-e                 — select a saved connection or enter params manually
                             Mode-line shows MySQL[root@127.0.0.1:3306/mydb]
4. C-c C-c                 — execute region, or the current statement/query at point
#+end_src

In ordinary =clutch-mode= or REPL buffers, =C-c C-e= keeps that generic
connect flow.  For SQLite files, prefer =M-x clutch-query-sqlite-file=, or
=M-x clutch-query-console= followed by RET with no matching connection and
backend =sqlite=; both open a connected SQL console rather than using the
database file buffer as the editor.  In query-console buffers, =C-c C-e=
reconnects the connection already associated with that console, without
reopening the global connection picker.
To switch to another saved or temporary connection, use =M-x clutch-query-console=.

For deeper troubleshooting, enable =M-x clutch-debug-mode=, reproduce the
failure, then inspect =*clutch-debug*=.  Enabling the mode starts a fresh
capture window and creates that dedicated buffer automatically.  It is the only
supported debug UI, and it shows problem records, generated/internal SQL when
relevant, recent redacted debug events, and JDBC stderr/debug payload when
available.

To activate =clutch-mode= automatically for =.sql= files, add to your config:

#+begin_src emacs-lisp
(add-to-list 'auto-mode-alist '("\\.sql\\'" . clutch-mode))
#+end_src

=.mysql= files activate =clutch-mode= automatically without any configuration.

Detailed key bindings, result-browser workflows, object navigation, Embark integration, transient menus, faces, timeouts, and customization now live in [[file:docs/interactive-client.org][docs/interactive-client.org]].

Common entry points:

** Unit tests (no server required)

=test/clutch-test.el= — UI layer, covering value formatting and cell rendering,
column width and single-page result layout, insert and edit buffer workflow
(TAB navigation, local validation, JSON sub-editor, enum/temporal completion),
staged-commit lifecycle (row-identity based edit/insert/delete, discard, commit
ordering), object workflow and schema refresh state, SQL keyword completion
(CAPF), and REPL behavior.

=test/clutch-db-test.el= — database backend layer, covering JDBC
connection and timeout mapping, query-timeout clamping, transaction
(commit/rollback/autocommit toggle), schema refresh, cursor pagination, driver
installation (Oracle, SQL Server, Redshift), agent JAR validation and lifecycle,
=clutch-db-live-p= identity check, row identity candidates, RPC timeout kill,
native backend adapter mapping (mysql / pg / sqlite), and cross-backend value
formatting and null handling.

The pure protocol packages (=mysql= and =pg=) and the Org-Babel bridge
(=ob-clutch=) now live in separate repositories and carry their own focused test
suites.

Default ERT runs intentionally skip live tests when database credentials are not
set.  Treat those runs as unit/regression coverage, not as proof that real
database workflows still work.  Test commands use =load-prefer-newer= so stale
=.elc= files cannot mask source changes during local verification.

** Native live tests

=test/run-native-live-tests.sh= runs the MySQL and PostgreSQL live suites against
local containers.  On Linux it prefers Podman when available; on macOS it
requires OrbStack-backed Docker unless explicitly overridden.  It covers both
the UI-level result workflows and backend-level native adapter tests:

#+begin_src sh
./test/run-native-live-tests.sh
#+end_src

The live runner validates real SELECT execution, pagination/sort/filter/count,
export collection, object describe metadata, staged insert/update/delete commit,
aggregate SELECT row-identity behavior, PostgreSQL =ctid= row locators, and MySQL
index-detail scoping.  It stubs only Emacs UI boundaries such as display-buffer
selection and confirmation prompts; query execution, metadata, staging SQL, and

* Roadmap

- AST-based SQL rewriting is deferred, not planned as a near-term feature.
  The current top-level clause scanner plus derived-table fallback covers known
  WHERE/filter/count/pagination cases across CTE, UNION, nested SELECT, and
  window-function queries.  Revisit a full parser only if concrete rewrite bugs
  or optimization features justify the added complexity.
